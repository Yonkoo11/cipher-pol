Protocol

How it works

Five steps. The ZK proof is generated client-side. The server never sees the deposit identity.

01
On-chain / public

Deposit into the privacy pool

Agent generates a random (secret, nullifier) pair, computes commitment = H(H(secret, nullifier), amount) via Poseidon2, and calls pool.deposit(H(secret, nullifier), amount). The deposit address is visible on-chain. The secret and nullifier never leave the agent.

02
Off-chain / private

Wait for the anonymity set to grow

Each additional deposit into the pool adds cover. Withdrawing immediately after your own deposit = zero privacy. With 50+ deposits between yours and the withdrawal, the deposit→payment link is computationally infeasible to brute-force.

03
Client-side / private

Generate a Groth16 proof

Agent fetches all Deposit events, reconstructs the 24-depth Poseidon Merkle tree, and proves: "I know (secret, nullifier) whose commitment is a leaf in the pool, and I authorize payment to recipient." Private inputs never leave the device. Public outputs: root, nullifierHash, recipient, amount.

04
HTTP / private

Pay via x402 — proof in a header

Agent hits the API. Gets a 402 Payment Required. Retries with X-Payment-Proof header containing only the ZK proof and public inputs. No txHash. No address. No link to the deposit.

05
On-chain / public

Server withdraws funds

Server checks public inputs (recipient, amount, no nullifier reuse), serves the request, then queues pool.withdraw(proof). The Groth16 verifier on Starknet (garaga-generated) verifies the BN254 proof. Server receives funds. Nullifier stored on-chain — no replay possible.

Guarantees

What privacy you actually get

No sugarcoating. Here's exactly what the ZK proof hides and what it doesn't.

✓ Hidden from everyone

  • Which deposit funded this payment
  • Depositor's secret and nullifier
  • The deposit transaction hash
  • Which leaf index in the Merkle tree
  • Any link between depositor and API consumer

⚠ Visible on-chain (v1)

  • Depositor's Starknet address (deposit event)
  • Deposit amount
  • API server received payment (withdrawal event)
  • Payment amount and timestamp (correlation risk)
Anonymity set matters. If your pool has 3 deposits total and you are one of them, there are only 3 candidates for any withdrawal. Wraith v1 is privacy-meaningful with 50+ deposits in the pool. Small sets dramatically narrow the candidate space.

Integration

Drop-in x402 payments

Agent generates the proof and retries — one await call.

typescript agent-payment.ts 
// 1. Set up the agent once
import { WraithAgent, PrivacyPoolsAdapter } from '@wraith/sdk';

const adapter = new PrivacyPoolsAdapter(starknetAccount, poolAddress);
const agent   = new WraithAgent(starknetAccount, adapter);

// 2. Deposit into the pool (once, reuse the note for many payments)
const { note } = await agent.deposit(1_000_000n, 'STRK'); // store note securely

// 3. Hit a 402-gated API — agent handles proof generation + retry
const response = await fetch('https://api.example.com/inference');

if (response.status === 402) {
  const challenge = parseChallenge(response);  // reads WWW-Authenticate

  // ZK proof generated client-side — 4-6s, never leaves device
  const proof = await adapter.generatePaymentProof(
    note,
    challenge.payTo,   // recipient = API server's Starknet address
    challenge.amount,
    { wasmPath, zkeyPath }
  );

  // Retry with X-Payment-Proof header — server verifies, queues withdrawal
  const paid = await fetch('https://api.example.com/inference', {
    headers: { 'X-Payment-Proof': buildPaymentHeader(proof) }
  });
  const result = await paid.json();
}

import { x402Middleware } from '@wraith/sdk/server';

app.use(x402Middleware({

Honesty

What this is and isn't

✓ What has been verified

  • 26/26 integration tests (proof pipeline)
  • 8/8 on-chain tests (devnet deposit + withdraw)
  • Poseidon2 matches Ekubo's circuit constants
  • garaga 0.15.3 calldata accepted by verifier
  • Local proof verification before serialization
  • x402 middleware and nullifier tracking

⚠ What has NOT been done

  • Production trusted setup (1-party, not MPC)
  • Third-party circuit audit
  • Testnet or mainnet deployment
  • Redis-backed nullifier set (in-memory only)
  • RapidSnark (proof is 4–6s not 100ms)
  • Lit Protocol tested against live network
